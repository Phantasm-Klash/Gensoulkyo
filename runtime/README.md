# runtime

Authoritative gameplay and service runtime modules.

- `core`: in-memory server domain model. It owns anonymous and external session mapping, player inventory, card upgrade cost/level changes, saved decks, active deck resolution, chest pools, chest ownership/cost/pity/opening audit records, queues, pre-match ticket cancellation, room-code lobbies, low-frequency lobby messages with idempotent ids and host-only announcements, server-validated deck/stage/character loadouts, heartbeat presence for sessions/tickets/matches, matches, player input validation, server hand/energy/cooldown card validation, deterministic stage-aware bullet simulation, active-card snapshots, server-validated mode actions for battle royale selection and Boss card transfer, Boss HP/damage/graze/hit projections, world-boss snapshots/daily attempts/global HP deduction, instance-boss clear/fail adjudication, snapshots, cursor-based match event logs, reconnect restore, match settlement, post-settlement rematch acceptance/new-match creation, server-generated replay audit records, rewards, tasks, events, leaderboards, and activity claim settlement.
- `security`: reusable migration-state security helpers. It currently owns the business envelope scaffold guard for version/seq/timestamp/nonce/op/key/tag validation, replay rejection, sanitized audit snapshots, SDK-neutral adapters from HTTP headers, Nakama RPC-style payloads, and Nakama WSS-style payloads into one `BusinessEnvelopeRequest`, and a standard-library `database/sql` audit sink adapter for the planned PostgreSQL audit table. It is intentionally not production X25519/AEAD/sign yet, but is shaped so HTTP, future Nakama RPC, and future business WSS can share one guard implementation.
- `storage`: optional standard-library database configuration and `.up.sql` migration helpers. The local service entrypoint registers the pgx `database/sql` driver as `pgx`; the runtime package itself remains driver-neutral.
- `httpapi`: standard-library HTTP adapter for the core service.
- `nakamaapi`: SDK-neutral Nakama Runtime adapter skeleton. It maps external Nakama user/session ids into core sessions, dispatches core business RPCs and business WSS-style messages from plain Go request structs, requires the shared business envelope on authenticated calls, records missing/invalid/replayed envelope audits through the shared guard, and keeps the real Nakama SDK binding as a thin outer layer. The current lobby MVP exposes room create/list/get/rules/join/leave/message plus matchmaking ticket and battle allocation/ticket orchestration; room rules snapshots publish server-owned ruleset and deck hashes without accepting client-authored combat or reward state.

The core package deliberately avoids HTTP, Nakama, and database dependencies so it can later be reused from Nakama RPC/match handlers or backed by PostgreSQL repositories. Security state that must survive transport migration lives outside the HTTP adapter. HTTP already uses the shared adapter; `nakamaapi` now proves the same guard path for RPC/WSS-style payloads without forcing a Nakama SDK dependency into the local MVP. `cmd/gensoulkyo_nakama` contains the optional `nakama` build-tagged SDK binding source. The SQL audit sink expects a caller-provided `*sql.DB`; `cmd/gensoulkyo` can provide one with `-database-url` and can run pending `.up.sql` migrations with `-migrate-up`.
