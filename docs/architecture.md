# Architecture

Gensoulkyo is planned as the authoritative server for Phantasm Klash online play.

The first implementation target is a self-hosted server that supports:

- player account creation and login without commercial platform dependencies;
- lobby and room creation;
- 1v1 match sessions;
- tick-based input ingestion;
- server-owned random seeds;
- validated card activation requests;
- match snapshots;
- replay metadata persistence.

The client submits intent. The server decides authoritative state.

## First Runtime Contracts

Nakama RPC IDs:

- `gensoulkyo.config.get`: returns current ruleset/config version, tick rate, snapshot interval, and deck limits.
- `gensoulkyo.profile.get`: returns authenticated user ID plus the persisted open-server wallet.
- `gensoulkyo.deck.save`: validates and stores a 20-card deck in PostgreSQL `player_decks`.
- `gensoulkyo.deck.list`: lists decks from PostgreSQL `player_decks`.
- `gensoulkyo.inventory.get`: returns persisted wallet and card inventory rows.
- `gensoulkyo.chest.pool.list`: lists currently enabled open-server chest pools.
- `gensoulkyo.chest.open`: charges server-side cost, rolls from the pool with a server seed, writes an audit row, and grants the resulting card.
- `gensoulkyo.room.create`: validates the host active deck and creates a short-code `duel` room.
- `gensoulkyo.room.join`: validates the guest active deck and joins an open room by code.
- `gensoulkyo.room.start`: host-only start that revalidates active decks, locks snapshots, and creates a `gensoulkyo.duel` match.
- `gensoulkyo.match.create`: validates active deck snapshots and creates an authoritative `gensoulkyo.duel` match.

Authoritative match handler:

- Match name: `gensoulkyo.duel`.
- Client op codes: `1` ready, `2` input packet, `3` reconnect request.
- Server op codes: `101` match start, `102` state snapshot, `103` match end.
- First simulation target: 30 ticks/second, snapshot every 3 ticks.
- Realtime joins are restricted to the participants locked into the match params by room start or direct match creation.
- Input packets are validated for monotonic tick/sequence, valid direction masks, valid card-slot range, and server-owned bomb resources. Invalid packets are discarded and audited as match events.
- The match-end snapshot includes server-computed result and reward summaries; clients must display this settlement instead of deriving rewards locally.

Persistence boundary:

- Match IDs generated by the runtime are PostgreSQL-compatible UUIDv4 strings.
- Match start writes `matches` and `match_players` with ruleset and server seed locked for the match.
- Room start stores both the durable PostgreSQL match UUID and Nakama realtime match ID so clients can join the realtime socket while persistence remains keyed by UUID.
- Runtime events are persisted to `match_events` for audit/replay scaffolding, including state-hash checkpoints.
- Match completion updates final player stats/results and inserts one `match_reward_settlements` row per player.
- Local open-server rewards are idempotent and credit `player_wallets`; Steam Inventory drops remain outside the open repository.

Deck boundary:

- Current open-alpha deck limits are 20 cards and at most 2 copies of the same card.
- Deck saves validate enabled card ownership from `player_card_inventory` joined to `cards`, including owned-copy counts.
- Saving an active deck clears other active decks for the same user in the same serializable transaction.
- Direct match creation and room start revalidate each participant's active deck and lock the resulting deck snapshot into match params.
- `match_players.deck_snapshot_json` stores that immutable snapshot at match start.

This first milestone intentionally implements deterministic movement/input scaffolding and audit-friendly snapshots, not full bullet-pattern or card-effect resolution. The economy milestone implements local open-server inventory and chest primitives only; commercial platform inventory, official drop rates, and Steam ownership checks remain outside this repository.
